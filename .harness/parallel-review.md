# 大规模并行审稿执行模式

> Hermes Agent 中使用 delegate_task 并行执行 6 维审稿的方法论。
> 审稿维度定义见 `审稿Agent.md` 和各 `skills/` 子模块。

## 何时使用并行模式

- 审查范围 > 10 章 → 子代理分片并行
- 审查范围 ≤ 9 章 → 主编 Agent 自行执行

## 分片策略

每子代理 ~9 章，按叙事边界切分（尊重 Part/卷边界，不机械等分）。

示例（27章）：
- Agent A: Ch1-9
- Agent B: Ch10-18
- Agent C: Ch19-27

## 子代理 context 模板

子代理无法读取 harness 文件——审查规范必须完整嵌入 context。

### 基础审查 6 维度清单（嵌入每个子代理 context）

```
维度1 科学自洽：
  ★致命: 终端拟人 | git改现实 | 非CTF渗透 | 多持有者
  ★严重: 蝴蝶效应无因果链 | 闪回=完整记忆 | 角色越界解释量子 | 量子术语在非专家内心
  ★建议: 规模不匹配 | 闪回频率异常

维度2 轻小说风格：
  ★致命: 尬煽(流泪/大段独白)
  ★严重: 翻译腔 | 千人一面 | 解释性对话 | 大情感大动作 | 角色OOC
  ★建议: 纯叙述>500字 | AI式礼貌 | 无呼吸点

维度3 日常主线交织：
  ★严重: 无功能日常 | 纯灌水
  ★建议: 连续纯日常/纯主线 | 重复功能 | 偏离5:5超10%

维度4 技术自洽：
  ★严重: 术语错误 | 命令错误 | 攻击链不合理 | 难度不匹配 | 一键hack | CTF比赛团队外协助
  ★建议: 虚构工具不合理 | git log格式 | daemon.md风格

维度5 语感：
  检测: 「因为」「不是…是…」解释旁白 | 过度精确 | 段尾升华 | 独白过长 | 排比

维度6 情节一致性：
  ★致命: 知识越界 | 信息边界违规
  ★严重: OOC | 时间线矛盾 | 伏笔矛盾
  ★建议: 残留引用 | 关系跳级
```

### 完整 context 模板（前段 Ch1-9 示例）

```
你是归零（科幻轻小说）的审稿 Agent。审查 Ch1-9。

项目路径：/home/kita/code/novel/归零/
章节路径：正文/第1章.md 到 正文/第9章.md
设定文件：.harness-project/constraints.md

审查规范（输出中文）：
按以下6维度逐章扫描，每个问题标注 [章节:段落位置] + 等级 + 规则依据 + 建议修复。

[上述 6 维度清单]

特别关注：[该分段已知的历史问题列表]

**关键：fix.md 验证指令** — fix.md 中标记「✅ 已修」的项目不可盲信。必须逐条读对应段落实际文本确认。如发现未实际修复，在报告中标注 [fix未执行] 并升级为 ★★ 严重。

输出格式：
每个维度独立一节，按 ★★★ / ★★ / ★ 分级列出。每个问题: [ChX LXX] 描述 → 规则依据 → 建议。最后给出汇总表（致命N处/严重N处/建议N处）。
```

中段和后段模板类似，调整特别关注项和大纲文件路径。

## 汇总

子代理全部返回后，主代理编译跨分段汇总表：

```
| 分段 | ★★★ | ★★ | ★ |
| Ch1-9    | 0 | 2 | 19 |
| Ch10-18  | 0 | 5 | 14 |
| Ch19-27  | 0 | 0 | 12 |
| 合计     | 0 | 7 | 45 |
```

提取所有 ★★ 严重项为优先修复清单。★ 建议项按模式归类（「因为旁白残留 ~20处」「结尾升华 3处」等），不逐条列举。

## 一致性检查（审稿后）

6 维审稿只检查每章内部质量，不检查跨章一致性。审稿+修复完成后必须再跑一轮大纲/设定 vs 正文的交叉检查。

### 两块并行

**子代理 A：大纲 vs 正文**
- 读取 `大纲/大纲.md` + `大纲/卷一后半细纲.md`
- 逐章比对：标题、核心事件、节拍设计、角色台词/反应
- 输出：偏差清单 + 严重度 + 修复建议

**子代理 B：设定 vs 正文**
- 读取 `.harness-project/constraints.md` + `笔记.md`
- 逐条检查：硬约束、角色知识边界、Want/Need/Flaw、设定修订项
- 输出：违规清单 + 设定规则 + 严重度 + 修复建议

### 修复：少数服从多数原则

当正文中多处引用不一致时：
1. grep 全卷所有引用 → 统计哪个版本占多数
2. 检查大纲/笔记.md 支持哪个版本
3. 将少数派章节统一为多数版本
4. 修复后再次 grep 验证全卷一致性

## 大纲管理原则

大纲是活文档，不是不可变的规约。正文和大纲出现偏差时，逐条判断：

- 正文写得比大纲好 → 更新大纲以匹配正文
- 大纲设计更合理 → 修改正文以匹配大纲
- 大纲内容按信息释放节奏后移 → 大纲标注「移至卷X」+ 理由
- 正向创作偏离 → 标记保留，同步更新大纲和伏笔登记表

**不要默认「大纲是对的」。逐条判断。**

## fix.md 管理

审查完成后，新发现的严重问题追加到 fix.md（标注日期和来源审查），已有问题更新修复状态。

**fix.md 标记不可信**：标记「✅ 已修」的项目可能在正文中并未实际修复。子代理和主编都必须逐条验证——读对应段落确认，不要信任 fix.md 的状态标记。

## Pitfalls 总汇

- 子代理无法读取 harness 的 SKILL.md——审查规范必须嵌入 context
- 不同子代理报告格式可能不一致——context 中明确输出模板
- 子代理可能漏读 fix.md——在 context 中写完整路径
- 子代理之间的发现可能重复——汇总时去重
- 单子代理 9 章 ~350s，3 并行 ~465s（受最慢子代理限制）
- **fix.md 标记不可信**（见上）
- **P0 模式密度统计**：「因为/不是…是…」复发不看「有无」看「密度」。单章 >10 处即超标
- **跨卷引用残留**：修改核心操作后必须 grep 全卷找出所有引用点统一修改
- **CTF 比赛伦理**：正式比赛禁止团队外协助。同队或赛后
- **操作一致性·少数服从多数原则**（见上）
- **审稿后必须做一致性检查**（见上）

## 词条（Glossary）创建

归零项目的词条系统——写作参考用的知识条目，放在项目 `词条/` 目录。

格式：每条 ~10 行，读者视角 glossary 风格，中文叙述 + 技术术语保留英文。

三类词条：
1. 小说专属：角色、物品、地点、机制、谜团
2. 科学概念：量子物理、因果律、时间悖论，需标注小说如何借用/改编
3. 计算机/黑客：CTF、渗透技术、网络协议、工具，实际技术 + 小说使用方式

维护：新增后更新 `词条/README.md` 索引，按类型分组。
